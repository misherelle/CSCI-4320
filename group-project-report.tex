\begin{abstract}
This project implements a massively parallel, Pokemon-themed battle territory simulation inspired by Conway's Game of Life. Each cell in a two-dimensional grid stores one of the 18 Pokemon elemental types, and each timestep updates territory ownership based on local neighbor support, type-effectiveness matchups, and rare pattern-triggered evolution events. The serial core defines the simulation rules and correctness tests, while the MPI version partitions the grid by rows and exchanges ghost rows at every iteration so that boundary cells are updated using neighboring rank data. The final project evaluates correctness, communication cost, computation cost, strong scaling, and weak scaling on AiMOS.
\end{abstract}

\begin{enumerate}

\item \section*{Introduction}

Our group built a Pokemon-themed battle territory simulation to study stencil-like parallel computation with nontrivial local interaction. The simulation is based on a two-dimensional cellular automaton, but instead of a binary alive/dead state, each cell stores one of the 18 official Pokemon types. At every timestep, nearby cells compete for control of the center cell using the Pokemon type-effectiveness chart, where attacks may have no effect, be not very effective, be normally effective, or be super effective.

This project fits the parallel programming goals of the course because each cell update depends on neighboring cells. The work is therefore not embarrassingly parallel: when the grid is split across MPI ranks, ranks must exchange halo rows before each update. Our implementation separates the serial simulation logic from the parallel execution layer so that correctness can be established first and then reused by MPI and future CUDA versions.

\item \section*{Serial Simulation Model and Correctness}

The serial model stores each cell as an unsigned 8-bit type ID. The type IDs are:
\[
\begin{aligned}
\text{Normal, Fire, Water, Electric, Grass, Ice, Fighting, Poison, Ground,} \\
\text{Flying, Psychic, Bug, Rock, Ghost, Dragon, Dark, Steel, Fairy.}
\end{aligned}
\]

The effectiveness table is represented as an $18 \times 18$ integer matrix. Rows are attacking types and columns are defending types. The values are:
\[
0 = \text{no effect}, \quad 50 = \text{not very effective}, \quad 100 = \text{normal}, \quad 200 = \text{super effective}.
\]

Each timestep uses the eight neighboring cells around a center cell. Same-type neighbors add support pressure to the center type. Different-type neighbors add attack pressure according to the attacker's effectiveness against the center type. If one opposing type exceeds the center's support by a threshold, the cell changes ownership to that type. If the center has enough same-type support, it remains stable. Rare evolution events are checked before ordinary takeover. For example, a Normal center with nearby Fire, Water, Grass, and Electric support evolves into Dragon.

Correctness is tested in \texttt{pokemon\_battle\_tests.c}. The tests check:
\begin{enumerate}
\item the full $18 \times 18$ type-effectiveness chart,
\item individual no-effect, half-effect, normal, and super-effective examples,
\item takeover by a stronger neighboring type,
\item stability from same-type support,
\item rare Dragon and Steel evolution patterns,
\item double-buffered timestepping so updates use only the old grid.
\end{enumerate}

\item \section*{Parallel Algorithm: MPI}

The current parallel version uses one-dimensional row decomposition. Rank 0 initializes the global grid and distributes contiguous row blocks with \texttt{MPI\_Scatterv}. Each rank stores its local rows plus two ghost rows: one above and one below. At every timestep, each rank exchanges boundary rows with its vertical neighbors using \texttt{MPI\_Sendrecv}. After the halo exchange, each rank updates only its owned rows, leaving ghost rows as communication buffers.

The pseudocode is:

\begin{lstlisting}[language=C]
MPI_Init()
rank, nprocs = MPI rank information
rank 0 initializes global grid
MPI_Scatterv global rows to each rank

for step in 0..steps-1:
    exchange top real row with rank-1
    exchange bottom real row with rank+1
    for each owned local row:
        for each column:
            next[cell] = pokemon_next_cell_bounded(...)
    swap current and next

MPI_Gatherv final rows to rank 0
rank 0 compares MPI result against serial result
MPI_Finalize()
\end{lstlisting}

The communication time is measured around halo exchange, and computation time is measured around the local cell update loop. The total time includes all iterations after the initial scatter and before the final gather. The rank-maximum time is reported because the slowest rank determines the parallel step completion time.

\item \section*{CUDA Extension Plan}

The code is structured so that the local row update loop can be replaced by a CUDA kernel in a future hybrid MPI+CUDA version. Each CUDA thread would update one owned cell. The device would store the current grid, next grid, and the 18 by 18 effectiveness table. MPI would still exchange ghost rows between ranks, while CUDA would accelerate the per-cell update inside each rank. The intended comparison is CPU-only MPI versus hybrid MPI+CUDA.

\item \section*{Analysis and Discussion}

The main quantities to analyze are total runtime, communication time, computation time, speedup, and efficiency. Strong scaling speedup can be computed as:
\[
S_p = \frac{T_1}{T_p}
\]
where $T_1$ is the one-rank runtime and $T_p$ is the runtime on $p$ ranks. Strong scaling efficiency is:
\[
E_p = \frac{S_p}{p}.
\]

For weak scaling, the ideal result is constant runtime as $p$ increases while the work per rank remains constant. If weak scaling time increases, likely causes include halo exchange overhead, synchronization imbalance, cache effects, and rank-to-rank load differences when rows are not perfectly divisible.

\item \section*{Conclusions and Future Work}

This project demonstrates a non-embarrassingly-parallel cellular automaton with a richer state space than Conway's Game of Life. The serial implementation establishes correctness through a full type-chart test and targeted rule tests. The MPI implementation introduces row-based domain decomposition and halo exchange, making the dependency between neighboring partitions explicit.

Future work includes adding a CUDA kernel for local cell updates, comparing CPU-only MPI against MPI+CUDA, expanding the initial-condition generator, adding checkpoint output through MPI I/O, and running larger experiments across multiple AiMOS nodes.
